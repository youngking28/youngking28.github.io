位运算小解

Posted on 2018-06-14

前言

笔者使用alibaba fastjson JSON.parseObject()方法将json转换为map时，map顺序变动（当有嵌套json时， linkedMap.class只能保证第一层级顺序）。从而尝试观看fastjson源码，发现了处使用了位运算做了骚操作，忍不住记录一下。

基本四个位运算举例

15 ＝ 1 1 1 1
14 ＝ 1 1 1 0

&后 ＝ 1 1 1 0 (与) 上下同为1， 则为1
|后 ＝ 1 1 1 1 (或) 上下有一个为1， 则为1
~后 ＝ 0 0 0 0 (非) 单值取反
^后 ＝ 0 0 0 1 (异或) 上下相同为0， 不同为1

<< 左移
>> 右移

ps：关注位运算时，尽量忽略掉他们的十进制值。

之前尽管看过，但还是没有应用经验。

fastjson中:

public final boolean isEnabled(int feature) { return (this.features & feature) != 0; }
乍一看，这在做什么？
继续看源码得知：
this.lexer.isEnabled(Feature.OrderedField)
看到这里，才知道这大概意思是，判断转换器中是否有Feature.OrderedField选项。（this.是笔者为了方便理解， 自行加上，源码中没有）
可是， 做一下 & 与运算并判断非0， 就可以得出转换器里存放Feature.OrderedField选项的结论了？
思考无解， 百度重新了解了一下四个位运算，同时看到有个人提到过， A |= B是把B标记在A中，(A & B) != 0是可以判断A里是否有标记B。
经过取值计算验证， 得不出这个结论。 那回头继续看阿里爸爸为什么是这么做。

Feature(){ mask = (1 << ordinal()); }

在重新了解位运算后， 再看到这段普通的构造函数。我就忽然明朗了。真的写的太漂亮了， 怪不得能用位运算做标记， 怪不得是阿里爸爸。
将1左移n位，每一位的1代表了一个Feature类型的启用。这将大大提升了 转换器你有哪些要求需要我注意的 这个操作的性能。从一堆的if判断中，变成了每一小个的位运算判断。

另记录一个小技巧：不用第三个变量迅速完成赋值交换。
a = a ^ b, b = b ^ a, a = a ^ b

Deploy webapp with idea and tomcat

Posted on 2018-05-23

idea如何结合tomcat，部署web程序。

下载和安装tomcat，这里就不赘述了。

先在idea中配置tomcat。
选择添加tomcat server

设置server名字和home地址


之后， 在idea的module setting(右键工程，选中open module setting)中，先对你的工程设置好相对应的文件夹类型，idea中有Sources（源文件夹），Tests（测试文件夹）， Resources（资源文件夹），TestResource（测试资源文件夹），Exclude（排除文件夹）五种。

其中，Sources中的java文件，会被编译生成class文件，输出到指定path下（在module中可以设置）。
Resources则是项目所需要的文件配置，将会被一起打包放入输出path下，strutcs.xml等框架配置文件必须存放，否则将不会被打包， eclipse则是会一起打包。

设置完成后，去Facets中，创建刚刚moudle下的web子包，其中设置好web.xml的路径， 以及web页面的文件夹路径（WebRoot或者web）。

下一步去Artifacts中，将上面所设置的web包打包成一个 web application expload， 其中需要在WEB-INF下的classes指定上面module的complie output， 这样module编译后的class将会被打包进这个artifacts中。

在这一步中，有时候可能会碰到，资源文件无法被编译（笔者尝试多种方案后，暂时没有找到原因和解决方案），这时可以在artifacts这一步中，添加玩module的complie output，同时将资源文件的文件夹添加进这个artifacts， 就可以了。

最后，在tomcat， deploy选项中，加入这个artifacts， 启动tomcat， 就完成了部署！

Posted on 2018-04-19

前言

redis为什么效率高？

通过了解了redis的文档， 得知redis的读取效率可以达到110000/s, 写入效率80000/s。
在被学长询问到：为什么redis单进程单线程还可以有这么高的效率？
我不假思索的答：完全使用内存，数据结构简单。
学长： :)

之后看了不少文档，加上了自己的理解，总结出共有三个特性使得redis效率惊人：

1、完全操作内存
2、数据结构简单
3、单线程多路I/O复用

无疑，完全操作内存， 这是redis效率惊人的原因之一。那单线程的redis和多线程memcached同样操作内存，却达到效率持平，这是为什么呢？这就是单线程多路I/O复用的功劳了。

memcached采用的是libevent（网络事件通知库），主线程不断的遍历询问网络端口，发现有读写操作，则丢给work线程（开好的子线程）去处理，这里的主要开销则是主线程的不断轮询。

redis则是自己包装了一套轻量级的select、epoll、evport、kqueues通用接口实现ae_event，减免了一些在自己框架中不必要的操作，在linux下默认使用epoll方法，不用进行轮询，不做无差别轮询复制，当事件产生即处理，处理线程复用，大大减少了io连接的效率流失，从而提升了效率，丝毫不比多线程的memcached差，甚至超越，又因为redis的方案处理多样化，故而被许多企业青睐。

PS：写到这里，有一个小疑问，那么当redis开启多线程，效率会不会更高？（之后的研究方向有了。。。）

更新：

据经验丰富同事(block)告知，redis操作应该颗粒度很小，处理方便迅速，这样一来，多线程之间的切换损耗有可能比处理操作的时间更长。
所以redis不用开启多进程，如果要多核cpu无法得到充分利用，可以选择开启多个单例。

redis的主要设计模式是什么？好在哪里？

其实仔细看完第一个问题的答案， 就对这个问题心中有数了。
redis主要的设计模式依旧是Reactor,但是它在基础调用中首选了epoll，这样就免去了很多无差别轮询，和句柄复制，从而大大提高了效率。

好在哪里？

速度快效率高，这已经不用再多说。单线程的代码简洁优雅，逻辑清晰自然，不存在多线程切换导致的性能损耗，不用有锁的问题继而更不会出现deadlock的可能。

题外

select、epoll、kqueues是三种常见的I/O模型实现方法，select是I/O multiplxing的实现，
而epoll和kqueues是高级的select，memcached使用的就是select，需要大量轮询。而epoll和kqueues则实现当有事件产生即返回产生事件的句柄，不用轮询，epoll是Lunix特有，而kqueue是unix中的实现方法。

evport是libevent的一个类库，产生事件接口。

Posted on 2018-04-17

前言

锁是什么, Java为什么需要锁？

如果要谈论起锁这件事情，那么一定要了解一下多进程并发。
简单理解就是 同时 在干某件事情。
这里的同时，就特别重要。例如说对于一个方法， 同时被5个地方调用，这时候就有了资源问题。

一起拿到了这个资源，那么该如何分配这个资源？
从根本上而言， 没有绝对的并发。时间颗粒度有极为细小的区分，只是这段方法或者这个资源还没有被上一个调用者使用完毕，就要被下一个调用者使用，这个时候就产生了并发现象，产生数据错乱，甚至死锁deadlock。

这就是为什么Java为什么需要锁的原因。

Java的锁是，对这个资源上锁，实质上是给这个资源一个标志，告诉所有人：你们这些人得按规矩使用这个资源，要排队！上一个还没用完，下一个不准使用！等着！

有哪些锁？

总体而言， 有两种锁机制。
乐观锁（Optimistic Locking）悲观锁（Perssimisitc Lock）
对于所有而言：
悲观锁：假定会发生并发冲突，屏蔽一切可能违反数据完整性的操作。
乐观锁：假设不会发生并发冲突

对于Java而言：
悲观锁是一个独占锁的概念，假设这个资源一定会发生并发问题，当一个资源不可用时， 强制让其他进程挂起，直到这个资源被释放，才唤醒其他等待进程，进行下一轮争夺资源和上锁。

乐观锁则是假设这个资源不会产生并发问题，进行了处理流程，但是要对结果进行检查，如果因为并发冲突问题而产生了错误，则重试原操作，直到成功并正确。

有哪些锁关键字？

Java中共有两个锁关键字synchronized volatile

synchronized
1.典型的悲观锁机制， 可以修饰方法、语句块，从而实现同一时刻只有一个线程能够执行
2.具有原子性、可见性

volatile
1.修饰变量，保证线程在每次使用变量的时候，都会读取变量修改后的最新的值。
2.具有可见性
3.volatile可以禁止进行指令重排。

指令重排是指处理器为了提高程序运行效率，可能会对输入代码进行优化，它不保证各个语句的执行顺序同代码中的顺序一致，但是它会保证程序最终执行结果和代码顺序执行的结果是一致的。指令重排序不会影响单个线程的执行，但是会影响到线程并发执行的正确性。

程序执行到volatile修饰变量的读操作或者写操作时，在其前面的操作肯定已经完成，且结果已经对后面的操作可见，在其后面的操作肯定还没有进行。

题外注
原子性：保持一致，要么不开始，开始必定要完成。
可见性：一旦修改，值对所有进程显示的都是最新值。
